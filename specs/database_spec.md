# Database Schema and Multi-Tenancy Specification

**Project:** BeatMind AI
**Version:** 1.0
**Date:** 2026-02-26
**Status:** Draft
**Stack:** PostgreSQL, Drizzle ORM, TypeScript

---

## 1. Overview

This specification defines the complete database schema, multi-tenancy strategy, indexing plan, partitioning approach, and ORM configuration for BeatMind AI -- a multi-tenant SaaS platform that provides real-time heart rate monitoring and AI coaching to gyms, pilates studios, cycling centers, and fitness academies.

### Scope

- 12 tables covering tenants, users, memberships, athletes, HR monitoring hardware, training sessions, real-time HR telemetry, session statistics, and AI coaching output.
- Shared-database, shared-schema multi-tenancy with application-layer tenant isolation.
- High-volume time-series ingestion for `hr_readings` (~7.5M rows/month at 10-gym scale).
- Drizzle ORM as the single data access layer with type-safe schema definitions.

### Design Principles

1. **Tenant isolation at the application layer.** Every tenant-scoped query passes through `withGymScope(gymId)`. No row-level security (RLS) is used in the initial release to keep operational complexity low.
2. **Append-only telemetry.** `hr_readings` is insert-heavy and read-heavy on recent windows. Schema and indexing optimize for these patterns.
3. **Minimal nullable columns.** Foreign keys that represent genuinely optional relationships (e.g., `athletes.user_id`, `sessions.trainer_id`) are nullable; everything else is `NOT NULL`.
4. **UUID primary keys everywhere** except `hr_readings`, which uses `BIGSERIAL` for write throughput and partition-friendliness.

## 2. Multi-Tenancy Strategy

### Approach: Shared Database, Shared Schema

All tenants (gyms) coexist in the same PostgreSQL database and the same set of tables. Every tenant-scoped table carries a `gym_id UUID NOT NULL` foreign key referencing `gyms.id`.

### Why This Approach

| Factor | Decision |
|--------|----------|
| Scale target (initial) | 10-50 gyms |
| Operational overhead | Minimal -- single database to back up, migrate, monitor |
| Query isolation | Application-layer `WHERE gym_id = $1` on every tenant query |
| Schema evolution | Single migration path for all tenants |
| Cost | Single PostgreSQL instance on existing VPS |

### Tenant Isolation Enforcement

Isolation is enforced exclusively at the application layer through a mandatory query-scoping utility:

```typescript
// lib/utils/gym-scope.ts
import { eq, and, SQL } from "drizzle-orm";

export function withGymScope<T extends { gym_id: typeof gyms.id }>(
  table: T,
  gymId: string
): SQL {
  return eq(table.gym_id, gymId);
}
```

Every server-side data access function that touches tenant-scoped data MUST include `withGymScope` in its `WHERE` clause. This is validated during code review. There is no database-level fallback (no RLS, no views) in v1.

### Cross-Tenant Access

Only the SuperAdmin role may query across tenants. SuperAdmin queries omit the `gym_id` filter and are restricted to routes under `/superadmin/*`, guarded by the `is_superadmin` flag on the `users` table.

## 3. Tables

### 3.1 `gyms` (Tenants)

The root tenant table. Each row represents one subscribing gym.

```sql
CREATE TABLE gyms (
  id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name                VARCHAR(255) NOT NULL,
  slug                VARCHAR(100) NOT NULL UNIQUE,
  address             TEXT,
  phone               VARCHAR(50),
  timezone            VARCHAR(50) NOT NULL DEFAULT 'America/Sao_Paulo',
  language            VARCHAR(10) NOT NULL DEFAULT 'pt-BR',
  clerk_org_id        VARCHAR(255) NOT NULL UNIQUE,
  tv_access_token     UUID NOT NULL DEFAULT gen_random_uuid(),
  subscription_status VARCHAR(20) NOT NULL DEFAULT 'active'
    CHECK (subscription_status IN ('active', 'suspended', 'cancelled')),
  subscription_plan   VARCHAR(100),
  max_athletes        INT NOT NULL DEFAULT 20,
  logo_url            TEXT,
  primary_color       VARCHAR(7) DEFAULT '#000000',
  secondary_color     VARCHAR(7) DEFAULT '#FFFFFF',
  created_at          TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  updated_at          TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now()
);
```

**Column Notes:**

| Column | Purpose |
|--------|---------|
| `slug` | URL-friendly identifier, used in public-facing routes. Must be unique across the platform. |
| `clerk_org_id` | Links this gym to its Clerk Organization. One-to-one mapping, enforced by `UNIQUE`. |
| `tv_access_token` | UUID token used to authenticate the TV dashboard (`/tv/[gymId]?token=TOKEN`). Can be regenerated by the gym owner. |
| `subscription_status` | Enum-like check constraint. `suspended` gyms can log in but cannot start sessions. `cancelled` gyms cannot log in. |
| `max_athletes` | Hard cap on active athletes. Enforced at the application layer during athlete creation. |
| `primary_color` / `secondary_color` | Hex color codes for white-label branding on the TV dashboard and athlete portal. |
| `timezone` | IANA timezone string. Used for session timestamps, report generation, and scheduling. |
| `language` | BCP 47 language tag. Used for AI coaching prompts and report localization. |

**Drizzle Schema:**

```typescript

  clerkOrgId: varchar("clerk_org_id", { length: 255 }).notNull().unique(),
  tvAccessToken: uuid("tv_access_token").notNull().default(sql`gen_random_uuid()`),
  subscriptionStatus: varchar("subscription_status", { length: 20 })
    .notNull()
    .default("active"),
  subscriptionPlan: varchar("subscription_plan", { length: 100 }),
  maxAthletes: integer("max_athletes").notNull().default(20),
  logoUrl: text("logo_url"),
  primaryColor: varchar("primary_color", { length: 7 }).default("#000000"),
  secondaryColor: varchar("secondary_color", { length: 7 }).default("#FFFFFF"),
  createdAt: timestamp("created_at", { withTimezone: true }).notNull().defaultNow(),
  updatedAt: timestamp("updated_at", { withTimezone: true }).notNull().defaultNow(),
});
```

### 3.2 `users`

Platform users linked to Clerk. A user may belong to zero or more gyms through `gym_memberships`.

```sql
CREATE TABLE users (

  is_superadmin   BOOLEAN NOT NULL DEFAULT false,
  created_at      TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  updated_at      TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now()
);
```

**Column Notes:**

| Column | Purpose |
|--------|---------|
| `clerk_user_id` | Clerk's unique user identifier (e.g., `user_2abc...`). Used to resolve the local user from Clerk's JWT. |
| `is_superadmin` | Platform-wide flag. Only the BeatMind AI operator has this set to `true`. Grants access to `/superadmin/*` routes. |
| `email` | Synced from Clerk on sign-up and profile update via webhook. Not used for authentication directly. |

### 3.3 `gym_memberships`

Junction table linking users to gyms with a role. A user can be an owner at one gym and a trainer at another, but cannot hold two roles at the same gym.

```sql
CREATE TABLE gym_memberships (
  id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id   UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  gym_id    UUID NOT NULL REFERENCES gyms(id) ON DELETE CASCADE,
  role      VARCHAR(20) NOT NULL CHECK (role IN ('owner', 'trainer', 'athlete')),
  is_active BOOLEAN NOT NULL DEFAULT true,
  created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  UNIQUE (user_id, gym_id)
);
```

**Column Notes:**

| Column | Purpose |
|--------|---------|
| `role` | Maps to Clerk Organization roles: `org:admin` -> `owner`, `org:trainer` -> `trainer`, `org:athlete` -> `athlete`. |
| `is_active` | Soft-disable a membership without deleting it. Inactive members cannot access the gym. |
| `UNIQUE(user_id, gym_id)` | A user can hold exactly one role per gym. To change roles, update the existing row. |

**Drizzle Schema:**

```typescript
// lib/db/schema/gym-memberships.ts
import { pgTable, uuid, varchar, boolean, timestamp, unique } from "drizzle-orm/pg-core";
import { users } from "./users";
import { gyms } from "./gyms";

export const gymMembershipRoleEnum = ["owner", "trainer", "athlete"] as const;
export type GymMembershipRole = (typeof gymMembershipRoleEnum)[number];

export const gymMemberships = pgTable(
  "gym_memberships",
  {
    id: uuid("id").primaryKey().defaultRandom(),
    userId: uuid("user_id")
      .notNull()
      .references(() => users.id, { onDelete: "cascade" }),
    gymId: uuid("gym_id")
      .notNull()
      .references(() => gyms.id, { onDelete: "cascade" }),
    role: varchar("role", { length: 20 }).notNull(),
    isActive: boolean("is_active").notNull().default(true),
    createdAt: timestamp("created_at", { withTimezone: true }).notNull().defaultNow(),
    updatedAt: timestamp("updated_at", { withTimezone: true }).notNull().defaultNow(),
  },
  (table) => [
    unique("uq_gym_memberships_user_gym").on(table.userId, table.gymId),
  ]
);
```

### 3.4 `athletes`

Extended profile for monitored individuals. An athlete may optionally be linked to a platform user account (`user_id`), but this is not required -- gyms can register athletes who never log in.

```sql
CREATE TABLE athletes (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID REFERENCES users(id) ON DELETE SET NULL,
  gym_id          UUID NOT NULL REFERENCES gyms(id) ON DELETE CASCADE,
  name            VARCHAR(255) NOT NULL,
  email           VARCHAR(255),
  phone           VARCHAR(50),
  age             INT,
  weight_kg       DECIMAL(5, 2),
  max_hr          INT NOT NULL DEFAULT 190,
  whatsapp_opt_in BOOLEAN NOT NULL DEFAULT false,
  is_active       BOOLEAN NOT NULL DEFAULT true,
  created_at      TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  updated_at      TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now()
);
```

**Column Notes:**

| Column | Purpose |
|--------|---------|
| `user_id` | Nullable. When an athlete creates a platform account and links it, this FK is populated. Enables the athlete portal. |
| `gym_id` | Tenant FK. An athlete belongs to exactly one gym. If a person trains at two gyms, they have two separate athlete records. |
| `max_hr` | Maximum heart rate used for HR zone calculation. Default 190 is a safe general estimate. Trainers can customize per athlete. |
| `weight_kg` | Used in calorie estimation formulas. Nullable because it may not be collected at registration. |
| `whatsapp_opt_in` | Explicit consent flag for receiving WhatsApp messages. Must be `true` before the system sends any messages. |
| `is_active` | Soft delete. Inactive athletes do not appear in band assignment or session monitoring. |

**Drizzle Schema:**

```typescript

  whatsappOptIn: boolean("whatsapp_opt_in").notNull().default(false),
  isActive: boolean("is_active").notNull().default(true),
  createdAt: timestamp("created_at", { withTimezone: true }).notNull().defaultNow(),
  updatedAt: timestamp("updated_at", { withTimezone: true }).notNull().defaultNow(),
});
```

### 3.5 `athlete_bands`

Persistent mapping between an athlete and a physical HR sensor band within a gym. The `sensor_id` is the ANT+ device number broadcast by the band. Each `sensor_id` must be unique per gym (two bands in the same gym cannot share a sensor ID).

```sql
CREATE TABLE athlete_bands (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  athlete_id  UUID NOT NULL REFERENCES athletes(id) ON DELETE CASCADE,
  gym_id      UUID NOT NULL REFERENCES gyms(id) ON DELETE CASCADE,
  sensor_id   INT NOT NULL,
  band_label  VARCHAR(50),
  is_active   BOOLEAN NOT NULL DEFAULT true,
  created_at  TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  UNIQUE (gym_id, sensor_id)
);
```

**Column Notes:**

| Column | Purpose |
|--------|---------|
| `sensor_id` | ANT+ device number. Integer in range 1-65535. Used to map incoming HR readings to the correct athlete. |
| `band_label` | Human-friendly label (e.g., "Band #3", "Blue-12"). Helps trainers identify physical bands. |
| `is_active` | When `false`, the mapping is ignored during sensor-to-athlete resolution. Allows re-assigning a sensor without deleting history. |
| `UNIQUE(gym_id, sensor_id)` | Within a gym, each sensor ID maps to exactly one band record. Cross-gym collisions are expected and allowed. |

**Drizzle Schema:**

```typescript
// lib/db/schema/athlete-bands.ts
import { pgTable, uuid, integer, varchar, boolean, timestamp, unique } from "drizzle-orm/pg-core";
import { athletes } from "./athletes";
import { gyms } from "./gyms";

export const athleteBands = pgTable(
  "athlete_bands",
  {
    id: uuid("id").primaryKey().defaultRandom(),
    athleteId: uuid("athlete_id")
      .notNull()
      .references(() => athletes.id, { onDelete: "cascade" }),
    gymId: uuid("gym_id")
      .notNull()
      .references(() => gyms.id, { onDelete: "cascade" }),
    sensorId: integer("sensor_id").notNull(),
    bandLabel: varchar("band_label", { length: 50 }),
    isActive: boolean("is_active").notNull().default(true),
    createdAt: timestamp("created_at", { withTimezone: true }).notNull().defaultNow(),
  },
  (table) => [
    unique("uq_athlete_bands_gym_sensor").on(table.gymId, table.sensorId),
  ]
);
```

### 3.6 `sessions`

A training session at a gym. Sessions can be started manually by a trainer or auto-started when the first sensor sends data.

```sql
CREATE TABLE sessions (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  gym_id           UUID NOT NULL REFERENCES gyms(id) ON DELETE CASCADE,
  trainer_id       UUID REFERENCES users(id) ON DELETE SET NULL,
  class_type       VARCHAR(100),
  status           VARCHAR(20) NOT NULL DEFAULT 'active'
    CHECK (status IN ('active', 'completed', 'cancelled')),
  started_at       TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  ended_at         TIMESTAMP WITH TIME ZONE,
  duration_seconds INT,
  athlete_count    INT NOT NULL DEFAULT 0,
  ai_summary       TEXT,
  created_at       TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now()
);
```

**Column Notes:**

| Column | Purpose |
|--------|---------|
| `trainer_id` | Nullable. Auto-started sessions may not have a trainer assigned. Can be set retroactively. |
| `class_type` | Free-text field (e.g., "spinning", "pilates", "cycling", "functional"). Used in reports and WhatsApp messages. |
| `status` | Lifecycle: `active` -> `completed` or `cancelled`. Only one session per gym should be `active` at a time (enforced at application layer). |
| `ended_at` | Set when session transitions to `completed` or `cancelled`. |
| `duration_seconds` | Computed as `ended_at - started_at` in seconds. Stored for query convenience. |
| `athlete_count` | Denormalized count of athletes who participated. Updated when session ends. |
| `ai_summary` | Post-session AI-generated summary text. Populated by the AI coaching service after session completion. |

**Drizzle Schema:**

```typescript

  aiSummary: text("ai_summary"),
  createdAt: timestamp("created_at", { withTimezone: true }).notNull().defaultNow(),
});
```

### 3.7 `hr_readings` (HIGH VOLUME)

The core telemetry table. Receives one row per athlete per reading interval (every ~5 seconds via HTTPS batch, or buffered from 1-second WebSocket stream). This is the highest-volume table in the system and requires partitioning.

```sql
CREATE TABLE hr_readings (
  id              BIGSERIAL PRIMARY KEY,
  session_id      UUID NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
  gym_id          UUID NOT NULL REFERENCES gyms(id) ON DELETE CASCADE,
  athlete_id      UUID NOT NULL REFERENCES athletes(id) ON DELETE CASCADE,
  sensor_id       INT NOT NULL,
  heart_rate_bpm  INT NOT NULL,
  hr_zone         INT NOT NULL CHECK (hr_zone BETWEEN 1 AND 5),
  hr_zone_name    VARCHAR(20) NOT NULL,
  hr_zone_color   VARCHAR(7) NOT NULL,
  hr_max_percent  DECIMAL(5, 2) NOT NULL,
  beat_time       TIMESTAMP WITH TIME ZONE NOT NULL,
  beat_count      INT NOT NULL DEFAULT 0,
  device_active   BOOLEAN NOT NULL DEFAULT true,
  recorded_at     TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now()
);
```

**Column Notes:**

| Column | Purpose |
|--------|---------|
| `id` | `BIGSERIAL` instead of UUID for sequential insert performance and smaller index size at scale. |
| `session_id` | Every reading belongs to exactly one session. Enables efficient per-session queries. |
| `gym_id` | Denormalized from `sessions` for direct tenant-scoped queries without joining `sessions`. |
| `sensor_id` | ANT+ device number. Denormalized for debugging and fallback if band mapping changes mid-session. |
| `heart_rate_bpm` | Raw BPM value from the ANT+ sensor. Integer range typically 30-220. |
| `hr_zone` | Computed HR zone (1-5) based on the athlete's `max_hr`. Stored to avoid recomputation. |
| `hr_zone_name` | Human-readable zone name: "Rest", "Fat Burn", "Cardio", "Hard", "Peak". |
| `hr_zone_color` | Hex color for the zone: Blue, Green, Yellow, Orange, Red. Stored for TV dashboard rendering without re-lookup. |
| `hr_max_percent` | `(heart_rate_bpm / athlete.max_hr) * 100`. Stored pre-computed. |
| `beat_time` | Timestamp from the ANT+ sensor event (as reported by the local agent). May differ from `recorded_at` if data was buffered. |
| `beat_count` | Cumulative heartbeat count from the ANT+ sensor since connection. Used for HRV analysis (future). |
| `device_active` | Whether the sensor was actively transmitting. `false` indicates a gap/timeout from the device. |
| `recorded_at` | Server-side timestamp when the row was inserted into PostgreSQL. Used for partitioning. |

**HR Zone Reference:**

| Zone | Name | % Max HR | Color |
|------|------|----------|-------|
| 1 | Rest | 50-59% | `#3B82F6` (Blue) |
| 2 | Fat Burn | 60-69% | `#22C55E` (Green) |
| 3 | Cardio | 70-79% | `#EAB308` (Yellow) |
| 4 | Hard | 80-89% | `#F97316` (Orange) |
| 5 | Peak | 90-100% | `#EF4444` (Red) |

**Drizzle Schema:**

```typescript
});
```

### 3.8 `session_athletes`

Aggregated statistics per athlete per session. Populated at session end (or incrementally during the session for live stats).

```sql
CREATE TABLE session_athletes (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id      UUID NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
  athlete_id      UUID NOT NULL REFERENCES athletes(id) ON DELETE CASCADE,
  sensor_id       INT,
  avg_hr          INT,
  max_hr          INT,
  min_hr          INT,
  calories        INT,
  time_zone_1_s   INT NOT NULL DEFAULT 0,
  time_zone_2_s   INT NOT NULL DEFAULT 0,
  time_zone_3_s   INT NOT NULL DEFAULT 0,
  time_zone_4_s   INT NOT NULL DEFAULT 0,
  time_zone_5_s   INT NOT NULL DEFAULT 0,
  joined_at       TIMESTAMP WITH TIME ZONE,
  left_at         TIMESTAMP WITH TIME ZONE,
  created_at      TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  UNIQUE (session_id, athlete_id)
);
```

**Column Notes:**

| Column | Purpose |
|--------|---------|
| `avg_hr` / `max_hr` / `min_hr` | Aggregate HR stats computed from `hr_readings` for this athlete in this session. |
| `calories` | Estimated calorie burn. Computed using the Keytel formula with athlete weight, age, sex, average HR, and duration. |
| `time_zone_1_s` .. `time_zone_5_s` | Total seconds spent in each HR zone. Five separate integer columns for query simplicity over a JSONB approach. |
| `joined_at` | Timestamp of the athlete's first HR reading in the session. |
| `left_at` | Timestamp of the athlete's last HR reading in the session. |
| `UNIQUE(session_id, athlete_id)` | One summary row per athlete per session. |

## 4. Indexes

All indexes are listed below with their DDL and rationale. Indexes are defined in a dedicated migration file to keep schema and index creation separate.

### 4.1 `hr_readings` Indexes (High Priority)

These indexes are critical because `hr_readings` is the highest-volume table.

```sql
-- Primary query pattern: fetch all readings for a session, ordered by time (TV dashboard, post-session reports)
CREATE INDEX idx_hr_readings_session_time
  ON hr_readings (session_id, recorded_at DESC);

-- SuperAdmin / gym-level queries: recent readings across a gym (health monitoring, debugging)
CREATE INDEX idx_hr_readings_gym_time
  ON hr_readings (gym_id, recorded_at DESC);

-- Athlete portal: fetch an athlete's HR history across sessions
CREATE INDEX idx_hr_readings_athlete
  ON hr_readings (athlete_id, recorded_at DESC);
```

**Rationale:** All three indexes use `recorded_at DESC` as the trailing column because queries almost always want the most recent data first. The session-time index is the most frequently hit during live sessions (AI coaching queries the last N minutes of a session). The gym-time index supports cross-session debugging. The athlete index supports the athlete portal's history and progress views.

### 4.2 `sessions` Indexes

```sql
-- Gym dashboard: list sessions for a gym, most recent first
CREATE INDEX idx_sessions_gym
  ON sessions (gym_id, started_at DESC);

-- Active session lookup: only one session per gym should be active at a time
-- Partial index keeps the index tiny and fast
CREATE INDEX idx_sessions_active
  ON sessions (gym_id, status)
  WHERE status = 'active';
```

**Rationale:** The partial index on active sessions is hit on every HR reading ingestion (to resolve which session to attach the reading to) and on every TV dashboard connection (to find the current session). It contains at most N rows where N = number of gyms.

### 4.3 `athletes` Index

```sql
-- Athlete listing for a gym, only active athletes
-- Used in band assignment, session monitoring, and gym dashboard
CREATE INDEX idx_athletes_gym
  ON athletes (gym_id)
  WHERE is_active = true;
```

**Rationale:** Partial index excludes deactivated athletes. Most queries (band assignment, session athlete resolution) only care about active athletes.

### 4.4 `athlete_bands` Index

```sql
-- Sensor-to-athlete resolution during HR reading ingestion
-- Given a gym_id and sensor_id, find the active band mapping
CREATE INDEX idx_athlete_bands_gym
  ON athlete_bands (gym_id, sensor_id)
  WHERE is_active = true;
```

**Rationale:** This index is hit on every incoming HR reading to resolve which athlete a sensor belongs to. The partial index ensures only active mappings are considered, preventing stale band assignments from interfering.

### 4.5 Additional Recommended Indexes

These are not specified in the PRD but are recommended based on query patterns:

## 5. Partitioning Strategy

### Target Table: `hr_readings`

`hr_readings` is the only table that requires partitioning at the projected scale. All other tables remain unpartitioned.

### Partitioning Method: Range on `recorded_at` (Monthly)

```sql
CREATE TABLE hr_readings (
  id              BIGSERIAL,
  session_id      UUID NOT NULL,
  gym_id          UUID NOT NULL,
  athlete_id      UUID NOT NULL,
  sensor_id       INT NOT NULL,
  heart_rate_bpm  INT NOT NULL,
  hr_zone         INT NOT NULL,
  hr_zone_name    VARCHAR(20) NOT NULL,
  hr_zone_color   VARCHAR(7) NOT NULL,
  hr_max_percent  DECIMAL(5, 2) NOT NULL,
  beat_time       TIMESTAMP WITH TIME ZONE NOT NULL,
  beat_count      INT NOT NULL DEFAULT 0,
  device_active   BOOLEAN NOT NULL DEFAULT true,
  recorded_at     TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  PRIMARY KEY (id, recorded_at)
) PARTITION BY RANGE (recorded_at);
```

**Important:** When using partitioning, the partition key (`recorded_at`) must be part of the primary key. The PK becomes `(id, recorded_at)`.

### Partition Creation

Partitions are created monthly, either manually or via a cron job:

```sql
-- Example: create partition for March 2026
CREATE TABLE hr_readings_2026_03 PARTITION OF hr_readings
  FOR VALUES FROM ('2026-03-01') TO ('2026-04-01');

-- Example: create partition for April 2026
CREATE TABLE hr_readings_2026_04 PARTITION OF hr_readings
  FOR VALUES FROM ('2026-04-01') TO ('2026-05-01');
```

### Automated Partition Management

A scheduled job (cron or pg_cron) creates the next month's partition on the 25th of each month:

```sql
-- Run on the 25th of each month
DO $$
DECLARE
  next_month_start DATE := date_trunc('month', now() + interval '1 month');
  next_month_end DATE := date_trunc('month', now() + interval '2 months');
  partition_name TEXT := 'hr_readings_' || to_char(next_month_start, 'YYYY_MM');
BEGIN
  EXECUTE format(
    'CREATE TABLE IF NOT EXISTS %I PARTITION OF hr_readings FOR VALUES FROM (%L) TO (%L)',
    partition_name,
    next_month_start,
    next_month_end
  );
END $$;
```

### Archival Strategy

| Age | Action |
|-----|--------|
| 0-3 months | Full resolution data, all indexes active |
| 3-6 months | Full resolution data, consider dropping non-essential indexes on older partitions |
| >6 months | Detach partition, compress (pg_dump), store in object storage (Cloudflare R2). Retain `session_athletes` summary data indefinitely. |

### Archival Process

```sql
-- 1. Detach the old partition
ALTER TABLE hr_readings DETACH PARTITION hr_readings_2025_06;

-- 2. Export to compressed file
-- pg_dump -t hr_readings_2025_06 --data-only --format=custom dbname > hr_readings_2025_06.dump

-- 3. Upload to Cloudflare R2
-- rclone copy hr_readings_2025_06.dump r2:beatmind-archives/hr-readings/

-- 4. Drop the detached table
DROP TABLE hr_readings_2025_06;
```

### Summary Tables

For long-term analytics and fast dashboard rendering, pre-aggregated summary tables are recommended (Phase 5):


### Assumptions

| Athletes per session | 20 (max) |
| Reading interval | 5 seconds (HTTPS batch) |
| Operating hours per day | 8 hours |
| Operating days per month | 26 days |
| Number of gyms (initial) | 10 |

### Calculation

```
Readings per athlete per hour   = 3600s / 5s = 720
Readings per athlete per day    = 720 * 8h = 5,760
Readings per athlete per month  = 5,760 * 26d = 149,760
Readings per gym per month      = 149,760 * 20 athletes = 2,995,200
Readings total per month        = 2,995,200 * 10 gyms = 29,952,000
```

**Revised estimate: ~30M rows/month at full capacity (10 gyms, 20 athletes each, 8h/day, 26 days/month).**

Note: The PRD estimate of ~7.5M rows/month assumes lower average utilization (not all gyms running at full capacity with all 20 athletes every operating hour). Realistic utilization is likely 25-50% of maximum, yielding **7.5M - 15M rows/month**.

## 8. Data Isolation: `withGymScope`

### Core Utility

All tenant-scoped queries MUST use `withGymScope` to inject the `gym_id` filter. This is the single enforcement point for tenant isolation.

```typescript
// lib/utils/gym-scope.ts
import { eq, and, type SQL, type SQLWrapper } from "drizzle-orm";
import type { PgColumn } from "drizzle-orm/pg-core";

/**
 * Returns a SQL condition that filters rows by gym_id.
 * MUST be included in the WHERE clause of every tenant-scoped query.
 *
 * @param gymIdColumn - The gym_id column reference from the target table
 * @param gymId - The authenticated user's current gym UUID
 * @returns SQL condition: table.gym_id = gymId
 *
 * @example
 * const athletes = await db
 *   .select()
 *   .from(schema.athletes)
 *   .where(
 *     and(
 *       withGymScope(schema.athletes.gymId, gymId),
 *       eq(schema.athletes.isActive, true)
 *     )
 *   );
 */
export function withGymScope(gymIdColumn: PgColumn, gymId: string): SQL {
  return eq(gymIdColumn, gymId);
}
```

### Usage Patterns

**Simple query:**


### Auth Context Resolution

The `gymId` used in `withGymScope` is resolved from the authenticated user's context, never from request parameters:

```typescript
// lib/auth/guards.ts
import { auth } from "@clerk/nextjs/server";
import { db } from "@/lib/db";
import * as schema from "@/lib/db/schema";
import { eq } from "drizzle-orm";

export async function requireGymContext(): Promise<{ userId: string; gymId: string; role: string }> {
  const { userId, orgId } = await auth();

  if (!userId || !orgId) {
    throw new Error("Unauthorized: missing user or organization context");
  }

  // Resolve gym from Clerk org ID
  const [gym] = await db
    .select({ id: schema.gyms.id })
    .from(schema.gyms)
    .where(eq(schema.gyms.clerkOrgId, orgId))
    .limit(1);

  if (!gym) {
    throw new Error("Unauthorized: gym not found for organization");
  }

  // Resolve user and membership
  const [user] = await db
    .select({ id: schema.users.id })
    .from(schema.users)
    .where(eq(schema.users.clerkUserId, userId))
    .limit(1);

  if (!user) {
    throw new Error("Unauthorized: user not found");
  }

  const [membership] = await db
    .select({ role: schema.gymMemberships.role })
    .from(schema.gymMemberships)
    .where(
      and(
        eq(schema.gymMemberships.userId, user.id),
        eq(schema.gymMemberships.gymId, gym.id),
        eq(schema.gymMemberships.isActive, true)
      )
    )
    .limit(1);

  if (!membership) {
    throw new Error("Unauthorized: no active membership");
  }

  return { userId: user.id, gymId: gym.id, role: membership.role };
}
```

## Appendix C: Foreign Key Cascade Behavior

| FK Relationship | ON DELETE |
|-----------------|-----------|
| `gym_memberships.user_id` -> `users.id` | `CASCADE` |

| `sessions.trainer_id` -> `users.id` | `SET NULL` |
| `hr_readings.session_id` -> `sessions.id` | `CASCADE` |
| `hr_readings.gym_id` -> `gyms.id` | `CASCADE` |
| `hr_readings.athlete_id` -> `athletes.id` | `CASCADE` |

| `session_athletes.athlete_id` -> `athletes.id` | `CASCADE` |
| `ai_coaching_messages.session_id` -> `sessions.id` | `CASCADE` |

| `hr_bands.gym_id` -> `gyms.id` | `CASCADE` |

Nullable FKs (`athletes.user_id`, `sessions.trainer_id`) use `SET NULL` to preserve the child record when the referenced user is deleted. All other FKs use `CASCADE` because child records are meaningless without their parent.

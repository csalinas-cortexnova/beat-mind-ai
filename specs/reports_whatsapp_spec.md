## 1. Overview

BeatMind AI auto-generates a post-session report for every athlete that participated in a training session. Each report contains heart rate statistics, time-in-zone breakdowns, estimated calories burned, an HR timeline chart, and an AI-generated session summary. Reports are rendered as mobile-first web pages branded with the gym's identity.

Athletes who have opted in to WhatsApp notifications receive a Twilio-powered template message two minutes after the session ends, containing key metrics and a tokenized link to their full report.

### Goals

- Zero manual work for trainers: reports are generated and delivered automatically.
- Reports serve as a retention and engagement tool for gyms (white-label branded).
- WhatsApp delivery ensures near-100% open rates compared to email.
- All data is derived from `hr_readings` captured during the live session.

### Dependencies

| Dependency | Purpose |
|------------|---------|
| `hr_readings` table | Raw heart rate data source |
| `session_athletes` table | Persisted per-athlete stats |
| `sessions` table | Session metadata and `ai_summary` |
| `athletes` table | Profile data (age, weight, phone, opt-in) |
| `gyms` table | Branding (logo, colors) and gym info |
| OpenAI API (gpt-4o-mini) | AI session summary generation |
| Twilio WhatsApp Business API | Message delivery |
| Recharts | Client-side chart rendering |

## 3. Report Generation Flow

### 3.1 Trigger

Report generation is triggered when a session transitions to `status = 'ended'`. This happens via:

1. **Manual end**: Trainer calls `POST /api/v1/gym/sessions/[id]/end`.
2. **Auto-end**: System detects 2 minutes of sensor inactivity and ends the session automatically.

Both paths set `sessions.status = 'ended'` and `sessions.ended_at = NOW()`, then invoke the report generation pipeline.

### 3.2 Pipeline Steps

```
Session ends
  |
  v
[Step 1] Calculate duration_seconds = ended_at - started_at
  |
  v
[Step 2] For each athlete in session (via hr_readings.athlete_id DISTINCT):
  |
  |-- [2a] Query hr_readings for this athlete + session
  |-- [2b] Calculate: avg_hr, max_hr, min_hr
  |-- [2c] Calculate: time_zone_1_s through time_zone_5_s
  |-- [2d] Calculate: calories (see Section 5)
  |-- [2e] Upsert into session_athletes
  |
  v
[Step 3] Update sessions.athlete_count = COUNT(DISTINCT athlete_id)
  |
  v
[Step 4] Generate AI summary:
  |-- Collect per-athlete stats (zone distribution, avg HR, duration)
  |-- Call OpenAI gpt-4o-mini with structured prompt
  |-- Store result in sessions.ai_summary
  |
  v
[Step 5] Schedule WhatsApp delivery (2-minute delay)
```

### 3.3 Per-Athlete Statistics Calculation

```typescript
// Pseudocode for stat calculation from hr_readings

interface AthleteSessionStats {
  avg_hr: number;          // Mean of all heart_rate_bpm values
  max_hr: number;          // Maximum heart_rate_bpm
  min_hr: number;          // Minimum heart_rate_bpm
  calories: number;        // Estimated (see Section 5)
  time_zone_1_s: number;   // Seconds in Zone 1
  time_zone_2_s: number;   // Seconds in Zone 2
  time_zone_3_s: number;   // Seconds in Zone 3
  time_zone_4_s: number;   // Seconds in Zone 4
  time_zone_5_s: number;   // Seconds in Zone 5
}
```

**Zone time calculation**: For each consecutive pair of `hr_readings` ordered by `recorded_at`, the time delta (in seconds) between them is attributed to the zone of the first reading. The zone is determined by comparing `heart_rate_bpm` against the athlete's `max_hr` using the thresholds defined in Section 9.

**SQL approach** (preferred for performance):

```sql
SELECT
  athlete_id,
  ROUND(AVG(heart_rate_bpm)) AS avg_hr,
  MAX(heart_rate_bpm) AS max_hr,
  MIN(heart_rate_bpm) AS min_hr
FROM hr_readings
WHERE session_id = $1 AND athlete_id = $2
GROUP BY athlete_id;
```

Zone time is calculated in application code using the ordered readings and time deltas between consecutive entries.

## 5. Calories Estimation

### 5.1 Primary Formula (Keytel et al., 2005)

Requires: average HR, age, weight (kg), gender, duration.

**Male:**
```
calories = (-55.0969 + 0.6309 * avgHR + 0.1988 * weightKg + 0.2017 * age) / 4.184 * durationHours
```

**Female:**
```
calories = (-20.4022 + 0.4472 * avgHR + 0.1263 * weightKg + 0.074 * age) / 4.184 * durationHours
```

Where:
- `avgHR` = average heart rate during the session (BPM)
- `weightKg` = athlete's weight in kilograms
- `age` = athlete's age in years
- `durationHours` = session duration in fractional hours (e.g., 45 minutes = 0.75)
- `4.184` = conversion factor from kJ to kcal

### 5.2 Fallback Formula

When age, weight, or gender data is missing, use a simplified MET-based estimation:

```
calories = (avgHR / 100) * durationMinutes * 4.5
```

This is a rough approximation that assumes ~4.5 kcal/min at 100 BPM. Accuracy is lower but avoids returning null.

### 5.3 Implementation

```typescript
// lib/hr/calories.ts

export interface CalorieInput {
  avgHr: number;
  durationSeconds: number;
  age?: number | null;
  weightKg?: number | null;
  gender?: 'male' | 'female' | null;
}

export function estimateCalories(input: CalorieInput): number {
  const { avgHr, durationSeconds, age, weightKg, gender } = input;
  const durationHours = durationSeconds / 3600;
  const durationMinutes = durationSeconds / 60;

  // Use primary formula if all data is available
  if (age != null && weightKg != null && gender != null) {
    let rawKcal: number;
    if (gender === 'male') {
      rawKcal =
        (-55.0969 + 0.6309 * avgHr + 0.1988 * weightKg + 0.2017 * age) /
        4.184 *
        durationHours;
    } else {
      rawKcal =
        (-20.4022 + 0.4472 * avgHr + 0.1263 * weightKg + 0.074 * age) /
        4.184 *
        durationHours;
    }
    return Math.max(0, Math.round(rawKcal));
  }

  // Fallback: simplified estimation
  const fallbackKcal = (avgHr / 100) * durationMinutes * 4.5;
  return Math.max(0, Math.round(fallbackKcal));
}
```

### 5.4 Data Requirements

| Field | Source | Required |
|-------|--------|----------|
| `avgHr` | Calculated from `hr_readings` | Yes |
| `durationSeconds` | `sessions.duration_seconds` | Yes |


If gender is NULL, the fallback formula is used.

### 6.4 Send Conditions

All three conditions must be true for a message to be sent:

1. `athletes.whatsapp_opt_in = true`
2. `athletes.phone IS NOT NULL AND athletes.phone != ''`
3. Phone number is valid (starts with `+` and contains 10-15 digits)

### 6.5 Auto-Send Timing

- Trigger: 2 minutes after `sessions.ended_at`.
- Implementation: Use a delayed job. In the initial implementation, use `setTimeout` with a 2-minute delay within the session-end handler. For production scaling, migrate to a proper job queue (e.g., BullMQ or a cron-based approach).
- Debounce: If a session is re-ended (edge case), cancel any pending WhatsApp job for that session before scheduling a new one.

### 6.6 Retry Policy

| Attempt | Timing | On Failure |
|---------|--------|------------|
| 1st | Immediate (at T+2min) | Retry once |
| 2nd (retry) | 30 seconds after first failure | Log error, skip athlete |

No further retries. Failed deliveries are logged with the Twilio error code and message SID for debugging.

### 6.7 Rate Limiting

- Maximum 1 WhatsApp message per athlete per session (enforced by checking a `whatsapp_sent` boolean or a sent timestamp on `session_athletes`).
- Twilio's own rate limits apply (typically 80 messages/second for WhatsApp Business API).

### 6.8 Schema Addition

Add to `session_athletes` table:

```sql
ALTER TABLE session_athletes ADD COLUMN whatsapp_sent_at TIMESTAMPTZ DEFAULT NULL;
ALTER TABLE session_athletes ADD COLUMN whatsapp_status VARCHAR(20) DEFAULT NULL;
-- Values: 'sent', 'failed', 'skipped', NULL (not attempted)
ALTER TABLE session_athletes ADD COLUMN report_token TEXT DEFAULT NULL;
```

## 9. HR Zones Definition

Heart rate zones are calculated as a percentage of the athlete's maximum heart rate (`athletes.max_hr`). Default max HR is 190 BPM when not configured.

| Zone | Name | Min % | Max % | Color | Hex |
|------|------|-------|-------|-------|-----|
| 1 | Recovery | 50% | 60% | Blue | `#3B82F6` |
| 2 | Fat Burn | 60% | 70% | Green | `#22C55E` |
| 3 | Cardio | 70% | 80% | Yellow | `#EAB308` |
| 4 | Hard | 80% | 90% | Orange | `#F97316` |
| 5 | Peak | 90% | 100% | Red | `#EF4444` |

### 9.1 Zone Boundary Rules

- Boundaries are inclusive on the lower end, exclusive on the upper end (except Zone 5 which is inclusive on both ends).
- HR below 50% of max HR is classified as Zone 1 (no "below zone" state).
- HR above 100% of max HR is classified as Zone 5.
- Example: Athlete with max HR 200. Zone 3 = 140 BPM to 159 BPM. Zone 4 = 160 BPM to 179 BPM.

### 9.2 Implementation

```typescript
// lib/hr/zones.ts

export const HR_ZONES = [
  { zone: 1, name: 'Recovery',  minPercent: 50,  maxPercent: 60,  color: '#3B82F6' },
  { zone: 2, name: 'Fat Burn',  minPercent: 60,  maxPercent: 70,  color: '#22C55E' },
  { zone: 3, name: 'Cardio',    minPercent: 70,  maxPercent: 80,  color: '#EAB308' },
  { zone: 4, name: 'Hard',      minPercent: 80,  maxPercent: 90,  color: '#F97316' },
  { zone: 5, name: 'Peak',      minPercent: 90,  maxPercent: 100, color: '#EF4444' },
] as const;

export const DEFAULT_MAX_HR = 190;

export function getZone(bpm: number, maxHr: number = DEFAULT_MAX_HR): (typeof HR_ZONES)[number] {
  const percent = (bpm / maxHr) * 100;

  if (percent >= 90) return HR_ZONES[4]; // Zone 5
  if (percent >= 80) return HR_ZONES[3]; // Zone 4
  if (percent >= 70) return HR_ZONES[2]; // Zone 3
  if (percent >= 60) return HR_ZONES[1]; // Zone 2
  return HR_ZONES[0]; // Zone 1 (includes below 50%)
}

export function calculateZoneTimes(
  readings: { heartRateBpm: number; recordedAt: Date }[],
  maxHr: number = DEFAULT_MAX_HR
): { zone1Seconds: number; zone2Seconds: number; zone3Seconds: number; zone4Seconds: number; zone5Seconds: number } {
  const zoneTimes = { zone1Seconds: 0, zone2Seconds: 0, zone3Seconds: 0, zone4Seconds: 0, zone5Seconds: 0 };

  for (let i = 0; i < readings.length - 1; i++) {
    const current = readings[i];
    const next = readings[i + 1];
    const deltaSeconds = (next.recordedAt.getTime() - current.recordedAt.getTime()) / 1000;

    // Cap delta at 30 seconds to handle gaps in data (sensor dropout)
    const cappedDelta = Math.min(deltaSeconds, 30);

    const zone = getZone(current.heartRateBpm, maxHr);
    const key = `zone${zone.zone}Seconds` as keyof typeof zoneTimes;
    zoneTimes[key] += cappedDelta;
  }

  return zoneTimes;
}
```

**Delta capping**: If the gap between two consecutive readings exceeds 30 seconds, cap the attributed time at 30 seconds. This prevents sensor dropouts or data gaps from inflating zone times.

## 11. Error Handling

### 11.1 Twilio API Failure

| Scenario | Behavior |
|----------|----------|
| Network timeout (Twilio unreachable) | Retry once after 30 seconds. If retry fails, set `whatsapp_status = 'failed'`, log error, continue to next athlete. |
| Twilio error response (4xx/5xx) | Retry once after 30 seconds. If retry fails, set `whatsapp_status = 'failed'`, log Twilio error code and message, continue to next athlete. |
| Rate limit (429) | Retry once after the `Retry-After` header value (or 60 seconds if not provided). If retry fails, log and skip. |

### 11.2 Missing or Invalid Phone Number

| Scenario | Behavior |
|----------|----------|
| `athletes.phone` is NULL or empty | Set `whatsapp_status = 'skipped'`, `reason = 'no_phone'`. No log entry at warning level (silent skip). |
| Phone number fails validation | Set `whatsapp_status = 'skipped'`, `reason = 'invalid_phone'`. Log at debug level. |

### 11.3 Opt-In Not Granted

| Scenario | Behavior |
|----------|----------|
| `athletes.whatsapp_opt_in = false` or NULL | Set `whatsapp_status = 'skipped'`, `reason = 'no_opt_in'`. Silent skip. |

### 11.4 Report Generation Failure

| Scenario | Behavior |
|----------|----------|
| No `hr_readings` for session | Log warning. Session remains valid. `session_athletes` not populated. No WhatsApp sent. |
| OpenAI API failure (AI summary) | Log error. Set `sessions.ai_summary = NULL`. Report renders without AI summary section. Stats are still calculated and stored. |
| Database write failure (stats) | Log error. Retry once. If retry fails, session still valid but report will show "Report generation in progress" until manually re-triggered. |
| Token generation failure | Log error. WhatsApp message not sent for that athlete (no URL to include). |

### 11.5 Report Page Errors

| Scenario | User Sees |
|----------|-----------|
| Invalid or expired token | "This report is no longer available. Contact your gym for a new link." |
| Session not found | 404 page |
| Data still generating | "Your report is being generated. Please check back in a few minutes." |
| Chart rendering error | Graceful degradation: show numeric stats without charts |
